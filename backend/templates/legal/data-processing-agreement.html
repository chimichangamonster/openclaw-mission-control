1. Purpose

This Data Processing Agreement ("DPA") governs the processing of personal data by the Processor on behalf of the Controller in connection with the VantageClaw platform services ("Services") as described in the Master Services Agreement between the parties.

2. Definitions

  • "Personal Data" — any information relating to an identified or identifiable individual, as defined under PIPEDA (Canada) and applicable provincial privacy laws.
  • "Processing" — any operation performed on Personal Data, including collection, storage, use, transmission, and deletion.
  • "Sub-processor" — any third party engaged by the Processor to process Personal Data on behalf of the Controller.
  • "Platform" — the VantageClaw multi-tenant AI business platform operated by the Processor.

3.2 Processing Activities

  • Processing data through AI language models for analysis, summarization, and response generation
  • Storing processed results in the Controller's isolated workspace
  • Transmitting data to authorized third-party LLM providers (see Section 6)

4. Data Isolation and Security

4.1 Tenant Isolation

  • Controller's data is stored in a dedicated, isolated gateway container with its own workspace, memory, and configuration
  • No data is shared between organizations on the Platform
  • Each organization has independent encryption keys for stored secrets
  • Database queries are scoped to the Controller's organization ID — cross-tenant data access is architecturally prevented

4.2 Encryption

  • At rest: AES-256-GCM authenticated encryption with HKDF-SHA256 key derivation for all sensitive fields (API keys, OAuth tokens, credentials)
  • In transit: TLS 1.2+ for all API communications; Tailscale WireGuard for internal network
  • Backups: GPG-encrypted with AES-256, SHA-256 checksums, 30-day retention

4.3 Access Controls

  • Role-based access control (RBAC) with 5 privilege levels: viewer, member, operator, admin, owner
  • Platform administrator access is separated into owner (full access) and operator (infrastructure only — cannot read client data)
  • All administrative access to Controller's data is audit-logged
  • API rate limiting (600 requests/minute per organization)

5. Platform Administrator Access

5.1 What the Processor CAN access

  • Organization metadata (name, slug, member count, feature flags)
  • Gateway health status and connectivity
  • Aggregate usage metrics (API calls, token consumption)
  • Audit logs for the Controller's organization
  • Error logs for troubleshooting (with Controller's permission)

5.2 What the Processor will NOT access without written consent

  • Email content or attachments
  • Chat conversation history
  • Decrypted API keys or OAuth tokens
  • Business documents, proposals, or reports
  • Calendar events or contact details
  • Financial records or transaction data

5.3 Emergency Access

In the event of a security incident, data corruption, or service outage, the Processor may access Controller's data to the minimum extent necessary to resolve the issue. The Controller will be notified within 24 hours of any such access, with a written description of what was accessed and why.

6.2 Self-Hosted Option

If the Controller requires that no data leave their infrastructure, the Processor offers an enterprise self-hosted deployment where the Platform runs entirely on the Controller's own servers. In this configuration, no sub-processors receive Controller data.

6.3 BYOK (Bring Your Own Key)

The Controller may provide their own LLM API key. When configured, LLM requests route directly from the Controller's gateway to their chosen provider, bypassing the Processor's API key and usage tracking.

7. Data Retention and Deletion

  • Controller's data is retained for the duration of the service agreement
  • Upon termination, the Processor will delete all Controller data within 30 days
  • The Controller may request data export at any time (gateway workspace archive)
  • Backup copies are automatically purged after the 30-day retention window
  • Audit logs are retained for 90 days after termination for compliance purposes

8. Breach Notification

  • The Processor will notify the Controller of any confirmed data breach within 72 hours of discovery (PIPEDA requirement)
  • Notification will include: nature of the breach, data affected, remediation steps taken, recommendations for the Controller
  • The Processor will cooperate with the Controller's breach response procedures

9. Controller Rights

  • The Controller may request a copy of all their data at any time
  • The Controller may request deletion of specific data records
  • The Controller may audit the Processor's security practices with 30 days' notice
  • The Controller may terminate this DPA and the service agreement at any time

10. Governing Law

This DPA is governed by the laws of the Province of Alberta, Canada, and the federal laws of Canada applicable therein, including PIPEDA.

11. Term

This DPA remains in effect for the duration of the Master Services Agreement and for 30 days after its termination (to allow for data deletion).
